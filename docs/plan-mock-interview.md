# 方案：重构为「问答驱动的模拟面试」

## 背景

当前 interview-assistant skill 的交互终点是生成 5 个 `.md` 文件（知识图谱、面试题、故事卡等）。经过实际测试和反思，发现这是一个重大的设计失误——用户的真正需求不是拿到一堆文件去自己复习，而是**直接跟 Claude 进行模拟面试练习**。

### 核心问题

- 生成的文件用户不会主动去读
- 真正有价值的是「被提问 → 尝试回答 → 获得反馈」的循环
- 文件应该是过程产物，存到 `.interview-docs/` 作为题库和复习材料，不是用户交互的重心

## 目标

将 skill 从「文件生成器」转变为「交互式面试教练」：

1. 脚本自动提取项目信息（不变）
2. Claude 内部分析生成知识图谱/题目/故事卡（用户无感）
3. **Claude 扮演面试官逐题提问，用户回答后立即给评分+分析+完美答案**
4. 面试结束后输出总结

## 方案详情

### 不改的部分

- `scripts/` 下所有脚本保持不变（session-extractor.mjs, code_analyzer.sh, run.sh）
- `references/01-03` prompt 模板保持不变（兼容性：模板内部不引用步骤编号，调用上下文从 5 步变 4 步不影响其行为）
- `.interview-docs/` 目录机制不变

### Phase 1: 新建 `references/04-mock-interview.md`

面试官行为准则 Prompt 模板，定义 Claude 作为面试官的行为规范。

#### 三层题目结构

| 层 | 来源文件 | 用途 |
|---|---|---|
| **题库** | `interview_questions.md`（02 生成） | D×5 道结构化题目（每决策：基础确认题×1、追问×3、扩展场景题×1）+ 评分要点（✅期待/❌警惕） |
| **答案参考** | `story_cards.md`（03 生成） | 完美答案的素材库，评判用户回答完整度 |
| **行为准则** | `04-mock-interview.md`（本文件） | 出题顺序、评分规则、追问策略、反馈格式 |

这样 02 是题库、03 是答案库、04 是裁判规则，三者职责清晰，没有废产物。

#### 出题顺序

按决策置信度排序（🟢→🟡），每个决策的题目推进流程：

```
基础确认题
  └─ A → Claude 自主判断是否追加扩展场景题（问题深度足够则追加，否则跳过）→ 跳下一决策
  └─ B → 追问 1（从该决策的 3 道追问中选一道未问过的）
           └─ A → 直接跳下一决策
           └─ B → 最多再追问 1 次 → 跳下一决策（B级总追问上限：2次）
           └─ C → 直接展示完美答案 → 跳下一决策（不换角度重问，B路径已追问过）
  └─ C → 换角度重问（从该决策 3 道追问中选一道未问过的，最多 1 次）→ 无论重问得分如何，展示完美答案 → 跳下一决策
```

> 扩展场景题只在 A 级时由 Claude 自主决定是否追加，B/C 跳过。C 路径先换角度重问再展示完美答案（避免透题），重问题来自 `interview_questions.md` 该决策的追问题，不即兴编造。

**内部进度追踪**（不输出给用户）：Claude 维护一个题目进度表 `[决策N-题M] ✅/⬜`，确保不重复提问。

#### 提问规则
- 每次只问一题，等用户回答后再继续
- **每个决策点完成后**主动询问：「已完成第 M 个决策（共 D 个），已问 N 题，继续还是结束？」（D 由脚本在 Step 2 末尾生成完题库后通过 `grep` 计数提取并注入对话，Claude 直接使用该值，不自行读文件数；N 为本场面试所有已提问题目总数，含基础确认题、追问、扩展场景题，由 Claude 在对话上下文中累加维护）
- **暖场**：开场固定问「请用 2-3 句话介绍你的项目，以及你在其中负责的部分」，用于降低紧张感。**暖场题不计入题目进度，也不评 A/B/C**。暖场回答用于校准项目上下文（如确认项目名称、核心模块）；若回答与知识图谱信息不一致，以知识图谱为准

#### 评分标准与反馈格式

评分依据 `interview_questions.md` 中每道题的「✅期待听到 / ❌警惕信号」。

| 等级 | 含义 | 反馈内容 | 追问策略 |
|------|------|----------|----------|
| A | 优秀，清晰有深度 | 亮点（不给完美答案，避免疲劳） | Claude 自主判断是否追加扩展场景题后跳下一决策 |
| B | 及格，但缺细节或逻辑 | 亮点 + 改进 + 完美答案（取自对应故事卡） | 同一决策追问，**最多追问 2 次**后强制跳下一决策 |
| C | 不及格，需要重新组织 | 换角度重问（最多 1 次，题来自该决策追问题）→ 无论得分如何，展示完美答案 → 强制跳下一决策 | 先重问后透题，避免完美答案削弱练习效果 |

> 完美答案取自 `story_cards.md` 对应卡片的 STAR 文本相关段落，不即兴编造。C 路径的换角度重问同样来自 `interview_questions.md` 该决策的追问题，不即兴编造。

#### 按决策分块读取（阅读顺序策略）

Claude Code 读文件时整文加载 context，无法做真正的延迟加载。分块读取是**阅读顺序指导**，不是 context 节省手段：
1. 开始时读入 `interview_questions.md` 和 `story_cards.md` 全文，主动比对两个文件的决策标题建立对应关系（不预设编号顺序完全一致）
2. 按置信度排序后，每进入一个决策时聚焦该决策的题目块和对应故事卡
3. 完成一个决策后在内部进度表标记，再推进下一个

#### 结束与总结

退出触发条件：用户在每决策询问时选择「结束」，或所有决策点完成（决策总数 D 在 Step 2 末尾生成完题库后读取，Step 3 开始时直接使用该值）。

**增量写入**：每个决策点完成后，Claude 指令脚本将该决策的评分追加写入 `.interview-docs/mock_interview_summary.md`（文件 I/O 交给脚本，不由 Claude 直接写文件）。**每次面试开始前**，脚本先执行 `echo -e "---\n### 面试记录 $(date '+%Y-%m-%d %H:%M')" >> .interview-docs/mock_interview_summary.md` 写入会话头（时间戳精确性由系统 `date` 命令保证，不由 LLM 生成）。**容灾范围**：仅保证评分记录不丢失；面试进度（当前决策、已问题目）存在 Claude 内存中，关闭终端后不可续接，重开后从头开始。全部结束后补充总评和建议。

面试结束后的完整总结格式：

```
## 模拟面试总结

**总题数**：X 题  
**评分分布**：A × N / B × N / C × N

### 强项
- ...

### 需要加强
- ...

### 建议复习（指向具体文件）
- 故事卡 N（XXX 决策）—— 你的回答缺少了取舍部分，参考 `.interview-docs/story_cards.md` 第 N 张
- 面试题 N-M（追问：XXX）—— 建议准备 `.interview-docs/interview_questions.md` 决策 N 的追问 M
```

### Phase 2: 重写 SKILL.md 工作流

从 5 步简化为 4 步。**重复练习分支**：Step 1 脚本跑完后检测 `.interview-docs/interview_questions.md` 是否存在——若存在，询问「检测到之前的分析结果，直接开始面试还是重新分析？」选择直接开始时跳过 Step 2（包括询问职级/JD），但**仍单独执行一次 `extract_decision_count.sh`** 从已有题库提取 D，确保进度提示中的 D 有值（职级/JD 只在生成阶段有用，跳过生成不影响 D 的提取）。

| 步骤 | 用户可见 | 说明 |
|------|----------|------|
| Step 1: 自动提取 | ✅ 看到脚本运行 | 运行 scripts，提取 session + code summary |
| Step 2: 分析与准备 | ⚡ 看到进度反馈 | 问职级+JD → 串行生成：01（知识图谱）→ 02（面试题）→ 03（故事卡）→ **脚本提取 D**（`grep` 计数题库决策数并输出到对话）→ 对用户说「分析完毕，共 D 个决策，开始面试？」 |
| Step 3: 模拟面试 | ✅ 核心交互 | **脚本写入会话头**（`date` 命令生成时间戳）→ 暖场题（不评分）→ 逐题提问 → 回答 → 评分+反馈（LLM）→ 每决策完成后**脚本追加写入**评分 → 询问继续/结束 |
| Step 4: 面试总结 | ✅ 看到总结 | 总评+建议+指向具体文件的复习推荐+存 mock_interview_summary.md |

### Phase 3: 更新 install.sh

`CLAUDE_CMD` heredoc 中的斜杠命令同步为 4 步流程描述。新增两个辅助脚本调用点：
- `scripts/extract_decision_count.sh`：Step 2 末尾提取 D，格式为 `echo "DECISION_COUNT=$D"`，供 SKILL.md 指令捕获并注入对话
- `scripts/write_session_header.sh`：Step 3 开始时写入会话头时间戳

### Phase 4: 更新 README.md

更新流程说明、产出文件列表、用户体验描述，体现「面试教练」而非「文档生成器」的定位。

## 设计决策

1. **题目数量**：不设上限，每个决策点完成后询问「继续还是结束？」作为自然退出点
2. **反馈风格**：A 级只给亮点；B/C 级给完整反馈（亮点+改进+完美答案）
3. **文件生成时机**：Step 2 按串行顺序（01→02→03）生成并给用户进度反馈，写文件动作用户可见但不需要阅读
4. **完美答案来源**：取自 `story_cards.md` 对应卡片的 STAR 文本，不即兴编造
5. **追问策略**：A→Claude 自主判断是否追加扩展场景题后跳下一决策 / B→追问（追问后 A 直接跳下一决策，B 最多再追问 1 次，C 视为该决策 C 处理）/ C→先换角度重问 1 次（题来该决策追问题，不即兴编造），无论得分展示完美答案后强制跳下一决策
6. **阅读顺序**：先读入两个文件全文并比对决策标题建立对应关系，再按置信度排序推进（不预设两文件决策顺序相同）
7. **停止机制**：每决策完成后主动询问，避免依赖"停"等歧义词触发
8. **总结质量**：推荐复习材料时指向具体文件和编号，让文件成为复习材料
9. **容灾**：每决策完成后脚本追加写入 `mock_interview_summary.md`（文件 I/O 交给脚本）；会话头时间戳由脚本 `date` 命令写入（精确，不依赖 LLM 生成）；仅保证评分记录不丢失，面试进度不可续接
10. **重复练习**：Step 1 后检测已有分析文件，选直接开始时跳过 Step 2（含职级询问），但仍单独执行 `extract_decision_count.sh` 从已有题库提取 D，确保进度提示有值；职级/JD 只在生成阶段有用，跳过生成不影响 D 的提取
11. **总结文件**：`mock_interview_summary.md` 存入 `.interview-docs/`
12. **暖场题**：不计入进度、不评 A/B/C，用于确认项目上下文；若回答与知识图谱不一致，以知识图谱为准
13. **决策总数**：D 由脚本在 Step 2 末尾生成完题库后通过 `grep` 计数提取并注入对话，Step 3 直接使用该值（交给脚本，不依赖 LLM 自行读文件数数）；N 由 Claude 在对话上下文中累加维护（无断点续接需求，不需持久化）

## 验证标准

1. 运行 `/interview-assistant` → 脚本完成 → 问职级 → 看到「正在分析...」进度 → 「分析完毕，共 D 个决策，开始面试？」（D 为脚本提取的实际数值，非占位符）→ 暖场题（不评分）
2. 暖场后第一道正式题来自 🟢 最高置信度决策的基础确认题
3. A 级回答 → 只收到亮点，直接下一题；B/C 级 → 收到完美答案（内容与 story_cards.md 对应决策一致）
4. C 路径：先换角度重问（题来该决策追问题，不即兴编造），无论重问得分如何展示完美答案，然后强制跳下一决策
5. 一个决策全部题目完成后收到「已完成第 M 个决策（共 D 个），已问 N 题，继续还是结束？」（D 为实际决策数，N 含所有题型）
6. 结束后总结包含具体故事卡/面试题编号的复习推荐，并存入 `mock_interview_summary.md`；中途关闭终端后重开，已完成决策的评分仍在文件中（但面试进度不可续接，需从头开始）
7. `.interview-docs/` 包含完整 6 个文件（见文件清单表）
8. 第二次运行时检测到已有分析结果，询问是否重新分析；选直接开始时完整跳过 Step 2（含职级询问）
