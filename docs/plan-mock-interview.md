# 方案：重构为「问答驱动的模拟面试」

## 背景

当前 interview-assistant skill 的交互终点是生成 5 个 `.md` 文件（知识图谱、面试题、故事卡等）。经过实际测试和反思，发现这是一个重大的设计失误——用户的真正需求不是拿到一堆文件去自己复习，而是**直接跟 Claude 进行模拟面试练习**。

### 核心问题

- 生成的文件用户不会主动去读
- 真正有价值的是「被提问 → 尝试回答 → 获得反馈」的循环
- 文件应该是过程产物，存到 `.interview-docs/` 作为题库和复习材料，不是用户交互的重心

## 目标

将 skill 从「文件生成器」转变为「交互式面试教练」：

1. 脚本自动提取项目信息（不变）
2. Claude 内部分析生成知识图谱/题目/故事卡（用户无感）
3. **Claude 扮演面试官逐题提问，用户回答后立即给评分+分析+完美答案**
4. 面试结束后输出总结

## 方案详情

### 不改的部分

- `scripts/` 下所有脚本保持不变（session-extractor.mjs, code_analyzer.sh, run.sh）
- `references/01-03` prompt 模板保持不变
- `.interview-docs/` 目录机制不变

### Phase 1: 新建 `references/04-mock-interview.md`

面试官行为准则 Prompt 模板，定义 Claude 作为面试官的行为规范。

#### 三层题目结构

| 层 | 来源文件 | 用途 |
|---|---|---|
| **题库** | `interview_questions.md`（02 生成） | 25 道结构化题目 + 评分要点（✅期待/❌警惕） |
| **答案参考** | `story_cards.md`（03 生成） | 完美答案的素材库，评判用户回答完整度 |
| **行为准则** | `04-mock-interview.md`（本文件） | 出题顺序、评分规则、追问策略、反馈格式 |

这样 02 是题库、03 是答案库、04 是裁判规则，三者职责清晰，没有废产物。

#### 出题顺序

按决策置信度排序（🟢→🟡），每个决策：
1. 先问 `interview_questions.md` 中对应决策的「基础确认题」
2. 根据评分决定是否进入追问（从该决策的 3 道追问中选）
3. A 级时可追加「扩展场景题」（加码挑战）

**内部进度追踪**（不输出给用户）：Claude 维护一个题目进度表 `[决策N-题M] ✅/⬜`，确保不重复提问。

#### 提问规则
- 每次只问一题，等用户回答后再继续
- 每 5 题主动询问：「已完成 N/25 题，继续还是结束？」（这也是自然退出点）
- **暖场**：第一题固定为「请用 2-3 句话介绍你的项目，以及你在其中负责的部分」，用于降低紧张感 + 校准后续题目难度

#### 评分标准与反馈格式

评分依据 `interview_questions.md` 中每道题的「✅期待听到 / ❌警惕信号」。

| 等级 | 含义 | 反馈内容 | 追问策略 |
|------|------|----------|----------|
| A | 优秀，清晰有深度 | 亮点（不给完美答案，避免疲劳） | 换到下一个决策点 |
| B | 及格，但缺细节或逻辑 | 亮点 + 改进 + 完美答案（取自对应故事卡） | 同一决策追问细节 |
| C | 不及格，需要重新组织 | 完美答案（取自对应故事卡） | 换角度重问同一知识点 |

> 完美答案取自 `story_cards.md` 对应卡片的 STAR 文本相关段落，不即兴编造。

#### 降低 Context 压力：按决策分块读取

不在 Step 3 开始时一次性读入所有 25 道题，而是：
1. 开始时只读 `interview_questions.md` 的决策目录（5 个标题）
2. 进入某个决策时，再读该决策的 5 道题 + `story_cards.md` 中对应的 1 张卡
3. 完成一个决策后记录进度，再读下一个

#### 结束与总结

退出触发条件：用户在每 5 题询问时选择「结束」，或所有决策题目用尽。

面试结束后输出总结，并保存到 `.interview-docs/mock_interview_summary.md`：

```
## 模拟面试总结

**总题数**：X 题  
**评分分布**：A × N / B × N / C × N

### 强项
- ...

### 需要加强
- ...

### 建议复习（指向具体文件）
- 故事卡 N（XXX 决策）—— 你的回答缺少了取舍部分，参考 `.interview-docs/story_cards.md` 第 N 张
- 面试题 N-M（追问：XXX）—— 建议准备 `.interview-docs/interview_questions.md` 决策 N 的追问 M
```

### Phase 2: 重写 SKILL.md 工作流

从 5 步简化为 4 步：

| 步骤 | 用户可见 | 说明 |
|------|----------|------|
| Step 1: 自动提取 | ✅ 看到脚本运行 | 运行 scripts，提取 session + code summary |
| Step 2: 分析与准备 | ⚡ 看到进度反馈 | 问职级+JD → 串行生成：01（知识图谱）→ 02（面试题）→ 03（故事卡）→ 对用户说「分析完毕，开始面试？」 |
| Step 3: 模拟面试 | ✅ 核心交互 | 暖场题 → 逐题提问 → 回答 → 评分+反馈 → 每5题询问继续/结束 |
| Step 4: 面试总结 | ✅ 看到总结 | 总评+建议+指向具体文件的复习推荐+存 mock_interview_summary.md |

### Phase 3: 更新 install.sh

`CLAUDE_CMD` heredoc 中的斜杠命令同步为 4 步流程描述。

### Phase 4: 更新 README.md

更新流程说明、产出文件列表、用户体验描述，体现「面试教练」而非「文档生成器」的定位。

| `references/04-mock-interview.md` | 新建 | 面试官行为准则 Prompt 模板 |

## 设计决策

1. **题目数量**：不设上限，每 5 题询问一次「继续还是结束？」作为自然退出点
2. **反馈风格**：A 级只给亮点（不给完美答案）；B/C 级给完整反馈（亮点+改进+完美答案）
3. **文件生成时机**：Step 2 按串行顺序（01→02→03）生成并给用户进度反馈，写文件动作用户可见但不需要阅读
4. **完美答案来源**：取自 `story_cards.md` 对应卡片的 STAR 文本，不即兴编造
5. **追问策略**：A→下一决策 / B→追问细节 / C→展示完美答案+换角度重问
6. **Context 压力**：按决策分块读取（每次只读当前决策的 5 道题 + 1 张故事卡）
7. **停止机制**：每 5 题主动询问，避免依赖"停"等歧义词触发
8. **总结质量**：推荐复习材料时指向具体文件和编号，让文件成为复习材料
9. **重复练习**：检测到 `.interview-docs/interview_questions.md` 已存在时，询问「直接开始面试还是重新分析？」
10. **总结文件**：`mock_interview_summary.md` 存入 `.interview-docs/`

## 验证标准

1. 运行 `/interview-assistant` → 脚本完成 → 问职级 → 看到「正在分析...」进度 → 「分析完毕，开始面试？」→ 暖场题
2. 暖场后第一道正式题来自 🟢 最高置信度决策的基础确认题
3. A 级回答 → 只收到亮点，直接下一题；B/C 级 → 收到完美答案（内容与 story_cards.md 一致）
4. 每 5 题收到「已完成 N/25 题，继续还是结束？」
5. 结束后总结包含具体故事卡/面试题编号的复习推荐，并存入 `mock_interview_summary.md`
6. `.interview-docs/` 包含完整 6 个文件（code_summary, extracted_decisions, project_knowledge_graph, interview_questions, story_cards, mock_interview_summary）
7. 第二次运行时检测到已有分析结果，询问是否重新分析
