### 故事卡 1：选择 BullMQ 而非 RabbitMQ 处理异步脚本生成任务

> 📊 置信度：🟢高  |  技术域：架构层

**口述文本（可直接背诵）：**

> 【S】我当时在做 Video Script Agent 的视频脚本生成功能，面临的情况是：用户提交一个短视频主题后，系统需要调用多个 LLM 进行多步骤处理，整个过程要1-3分钟。我们既不能让用户同步等待，又要保证任务可靠执行，还需要支持优先级队列和自动重试。
>
> 【T】当时需要解决的核心问题是：选择一个合适的任务队列中间件，来处理这些耗时的异步任务。如果不解决，用户会长时间等待无反馈，或者任务失败后无法自动恢复。
>
> 【A】我考虑过 RabbitMQ 和 BullMQ 两个方案。RabbitMQ 的好处是功能全面、企业级特性成熟，但问题在于需要额外部署和维护一个中间件，团队熟悉度不够。BullMQ 基于 Redis，我们项目已经在用 Redis 做缓存，好处是集成简单、内置了重试和延迟队列功能。最终选了 BullMQ，核心原因是：1）项目规模中等，不需要 RabbitMQ 的复杂路由功能；2）团队更熟悉 Redis，降低运维成本；3）开发速度快，两周就实现了完整队列系统。
>
> 【R】上线后验证了 BullMQ 能稳定处理每天约1000个脚本生成任务，失败率低于1%。如果重来，我会更早引入队列监控面板，因为 BullMQ 自带的监控功能比较基础，我们后来花了额外时间搭建 Prometheus 监控。

**可能的追问及应对要点**：
- Q：如果任务量增长10倍，BullMQ 还能胜任吗？
  A：需要重新评估。单机 Redis 有内存限制，可能需要升级到 Redis 集群，或者考虑迁移到 Kafka。关键指标是 Redis 内存使用率和任务堆积情况。

### 故事卡 2：设计四级 Agent 工作流提升脚本生成质量

> 📊 置信度：🟢高  |  技术域：Agent层

**口述文本（可直接背诵）：**

> 【S】我当时在优化 Video Script Agent 的脚本生成质量，面临的情况是：最初使用单个 LLM Agent 一次性生成完整脚本，结果质量不稳定，而且一旦某个部分出问题，整个脚本都要重做，调试起来非常困难。
>
> 【T】当时需要解决的核心问题是：如何提高脚本生成的可控性和可调试性。如果不解决，用户会收到质量参差不齐的脚本，我们团队也无法有效优化生成效果。
>
> 【A】我考虑过两个方案：一是优化单个 Agent 的 prompt 工程，让它更智能；二是拆分成多个专用 Agent。优化单个 Agent 的好处是架构简单，但问题在于 prompt 会变得极其复杂，而且错误难以定位。最终选了多 Agent 方案，拆分成4个专用 Agent：主题分析 → 结构规划 → 对话生成 → 分镜建议。核心原因是：1）每个 Agent 职责单一，prompt 可以针对性优化；2）可以独立测试和调试每个环节；3）未来可以并行执行提升速度。
>
> 【R】拆分后，脚本质量评分从平均3.2分提升到4.1分（5分制），调试时间减少了70%。如果重来，我会更早定义 Agent 间的数据接口规范，初期我们接口设计不够严谨，导致后期重构了一些代码。

**可能的追问及应对要点**：
- Q：Agent 之间如何保证数据一致性？
  A：我们使用 TypeScript interface 严格定义每个 Agent 的输入输出格式，并在运行时做数据验证。关键是在"结构规划 Agent"输出后，会有一个质量评估步骤，不合格的大纲不会进入后续环节。

### 故事卡 3：实现二级缓存策略降低 LLM 调用成本

> 📊 置信度：🟡中  |  技术域：性能层

**口述文本（可直接背诵）：**

> 【S】我当时在优化 Video Script Agent 的性能和成本，面临的情况是：LLM API 调用既慢又贵，每次生成脚本要30-60秒，成本约0.1元。而且我们发现用户经常生成相似主题的脚本，比如"职场沟通技巧"和"办公室沟通方法"其实可以复用结果。
>
> 【T】当时需要解决的核心问题是：如何减少重复的 LLM 调用，同时保证用户感知的响应速度。如果不解决，随着用户量增长，API 成本会失控，用户体验也会因为等待时间而下降。
>
> 【A】我考虑过几种缓存方案：一是只做内存缓存，响应最快但容量有限；二是只做 Redis 缓存，容量大但每次都要网络IO；三是二级缓存结合两者。内存缓存的好处是纳秒级读取，问题在于服务器重启就丢失。Redis 缓存的好处是持久化，问题在于有网络延迟。最终选了二级缓存：高频主题（5分钟内访问过的）放内存，TTL 5分钟；所有结果都存 Redis，TTL 24小时。核心原因是：1）80%的请求集中在20%的热点主题，内存缓存能覆盖；2）Redis 保证数据不丢失；3）整体命中率能达到65%。
>
> 【R】上线后，平均响应时间从45秒降到15秒，API 成本降低了60%。如果重来，我会更精细地调整 TTL 策略，初期我们统一用24小时，后来发现有些时效性强的主题应该缩短缓存时间。

**可能的追问及应对要点**：
- Q：如何判断两个主题"相似"？
  A：我们使用了 embedding 技术，把用户输入的主题文本转换成向量，然后计算余弦相似度。设定阈值0.85，超过就认为是相似主题。这个阈值是通过历史数据测试得出的平衡点。

### 故事卡 4：使用 WebSocket 实现实时进度展示

> 📊 置信度：🟢高  |  技术域：前端层

**口述文本（可直接背诵）：**

> 【S】我当时在设计 Video Script Agent 的用户体验，面临的情况是：脚本生成要1-3分钟，用户提交后页面就卡住了，不知道进度如何，有些用户会反复刷新或重复提交，造成服务器压力。
>
> 【T】当时需要解决的核心问题是：如何让用户实时了解任务进度，减少等待焦虑。如果不解决，用户流失率会很高，而且重复提交会导致资源浪费。
>
> 【A】我考虑过两种方案：一是前端定时轮询 API，二是使用 WebSocket 实时推送。轮询的好处是实现简单、兼容性好，但问题在于网络开销大、实时性差（最快1秒更新一次）。WebSocket 的好处是真正的实时推送、网络开销小，但问题在于实现复杂、要处理连接稳定性。最终选了 WebSocket，核心原因是：1）任务状态变化频繁（分析中→规划中→生成中→完成），需要高实时性；2）预计用户等待时间长，实时反馈能显著提升体验；3）未来可能扩展更多实时功能。
>
> 【R】上线后，用户重复提交率从25%降到8%，页面停留时间增加了40%。如果重来，我会更早考虑移动端优化，初期我们主要针对桌面端，后来发现移动端网络不稳定，WebSocket 容易断开，又补充了更激进的重连策略。

**可能的追问及应对要点**：
- Q：WebSocket 连接数有限制吗？如何支撑大量用户？
  A：单机 Node.js 大概能支撑6万左右连接，我们目前用户量远不到这个数。如果真的需要支撑10万+用户，我们会考虑使用 Socket.IO 集群，或者用专业的 WebSocket 服务如 Pusher。

### 故事卡 5：选择 Zustand 管理前端复杂状态

> 📊 置信度：🟡中  |  技术域：数据层

**口述文本（可直接背诵）：**

> 【S】我当时在构建 Video Script Agent 的前端架构，面临的情况是：前端需要管理多种复杂状态，包括任务队列、实时进度、用户偏好、生成历史等，而且团队规模小，需要快速上手。
>
> 【T】当时需要解决的核心问题是：选择一个合适的状态管理方案，既要功能强大，又要学习成本低。如果不解决，状态管理混乱会导致代码难以维护，新成员上手慢。
>
> 【A】我考虑过 Redux、Context API 和 Zustand。Redux 的好处是功能全面、生态成熟，但问题在于样板代码多、学习曲线陡。Context API 的好处是 React 原生、简单场景够用，但问题在于性能差、不适合复杂状态。Zustand 的好处是轻量、TypeScript 友好、API 直观。最终选了 Zustand，核心原因是：1）项目规模中等，不需要 Redux 的复杂中间件；2）团队 TypeScript 经验丰富，Zustand 的类型支持很好；3）开发速度快，两天就重构了所有状态逻辑。
>
> 【R】使用 Zustand 后，状态相关代码量减少了40%，新成员平均2天就能上手状态管理。如果重来，我会更严格地组织 store 结构，初期我们按功能模块划分 store，后来发现有些跨模块状态不好处理，又做了一次重构。

**可能的追问及应对要点**：
- Q：如果项目扩大，Zustand 还能满足需求吗？
  A：需要评估。如果状态复杂度显著增加，可能会遇到组织性问题。我们的预案是：1）引入更严格的 store 分层规范；2）考虑迁移到 Redux Toolkit，它有更成熟的中间件生态；3）关键是要有明确的迁移标准和时机判断。
