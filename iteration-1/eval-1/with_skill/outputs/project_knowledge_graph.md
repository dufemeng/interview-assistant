## 2. 关键架构决策清单

### 决策 1：采用 BullMQ 而非 RabbitMQ 处理异步任务
- **决策描述**: 选择 BullMQ（基于 Redis）作为任务队列，而非 RabbitMQ 或 Kafka
- **背景**: 项目需要处理视频脚本生成的异步任务，这些任务耗时长（LLM 调用）、需要重试机制、优先级队列
- **取舍**:
  - BullMQ 优势：与 Redis 集成简单、内置重试/延迟队列、适合中小规模实时任务
  - RabbitMQ 优势：更成熟的企业级特性、复杂路由规则
  - 选择 BullMQ 原因：项目规模中等、开发团队熟悉 Redis、无需复杂消息路由
- **置信度**: 🟢 高（Session 有明确讨论 + 代码有完整实现）

### 决策 2：设计四级 Agent 工作流而非单 Agent
- **决策描述**: 将视频脚本生成拆分为 4 个专用 Agent（主题分析 → 结构规划 → 对话生成 → 分镜建议）
- **背景**: 单 Agent 难以处理复杂的多步骤创意生成任务，需要模块化设计便于调试和优化
- **取舍**:
  - 单 Agent 方案：简单、上下文连贯，但难以定位问题、无法并行处理
  - 多 Agent 方案：复杂、需要设计数据传递接口，但可独立优化、支持并行、易于调试
  - 选择多 Agent 原因：项目需要高质量输出、可维护性重要、未来可能扩展更多 Agent
- **置信度**: 🟢 高（Session 详细讨论工作流设计 + 代码实现 4 个 Agent）

### 决策 3：实现二级缓存策略（内存 + Redis）
- **决策描述**: 为 LLM 生成结果设计二级缓存：内存缓存（高频主题，TTL 5分钟）+ Redis 缓存（所有结果，TTL 24小时）
- **背景**: LLM 调用成本高、响应慢，相似主题的脚本可以复用，提升用户体验
- **取舍**:
  - 无缓存：每次请求都调用 LLM，成本高、响应慢
  - 单级缓存：实现简单，但难以平衡内存使用和缓存命中率
  - 二级缓存：实现复杂，但能优化不同访问频率的数据
  - 选择二级缓存原因：项目有明确的热点主题模式、需要平衡性能和资源使用
- **置信度**: 🟡 中（Session 讨论缓存策略，但代码实现比 Session 描述更复杂，包含语义相似度匹配）

### 决策 4：使用 WebSocket 而非轮询展示实时进度
- **决策描述**: 前端通过 WebSocket 连接后端，实时接收任务状态更新，而非定时轮询 API
- **背景**: 脚本生成任务耗时长（1-3分钟），用户需要实时了解进度（当前步骤、预计剩余时间）
- **取舍**:
  - 轮询方案：实现简单、兼容性好，但网络开销大、实时性差
  - WebSocket 方案：实现复杂、需要处理连接稳定性，但实时性好、网络开销小
  - 选择 WebSocket 原因：任务状态变化频繁、需要高实时性、用户等待体验重要
- **置信度**: 🟢 高（Session 讨论实时性需求 + 代码完整实现 WebSocket 模块）

### 决策 5：选择 Zustand 而非 Redux 进行前端状态管理
- **决策描述**: 前端使用 Zustand 管理客户端状态，而非 Redux 或 Context API
- **背景**: 前端需要管理复杂的任务状态、用户偏好、UI 状态，需要轻量但强大的状态管理
- **取舍**:
  - Redux：功能强大、生态成熟，但样板代码多、学习曲线陡
  - Context API：React 原生、简单场景适用，但性能问题、不适合复杂状态
  - Zustand：轻量、TypeScript 友好、学习曲线平缓
  - 选择 Zustand 原因：项目规模中等、团队熟悉度、TypeScript 集成好
- **置信度**: 🟡 中（Session 只提到需要状态管理，未具体讨论选型；代码实现使用 Zustand）

### 决策 6：使用 Prisma 而非 TypeORM 作为 ORM
- **决策描述**: 后端使用 Prisma 作为 ORM 工具，而非 TypeORM 或 Sequelize
- **背景**: 需要与 PostgreSQL 交互，管理复杂的脚本数据模型（用户、任务、脚本版本、反馈）
- **取舍**:
  - TypeORM：NestJS 官方推荐、功能全面，但复杂查询类型支持有限
  - Prisma：类型安全优秀、迁移工具完善、查询 API 直观
  - 选择 Prisma 原因：团队重视类型安全、开发体验好、迁移工具完善
- **置信度**: 🔴 低（Session 完全未讨论 ORM 选型，纯代码推断）
